01
WEEK ONE · DAYS 1–7

Foundation
MVC + EF Core + DB

You already know CRUD — now we structure it properly. Build the habit grid, design the database schema, and wire up real data flow.

📘 LEARN FIRST 3 days
MVC Project Structure Day 1

Understand how Controllers → Actions → Views work. A Controller is a C# class, each method is an Action, Razor (.cshtml) files are your Views. Study the folder structure: Controllers/, Models/, Views/, wwwroot/. Learn how routing maps URL /Habits/Index to HabitsController.Index().

Resources → Microsoft Learn: MVC Overview Tim Corey YouTube: ASP.NET MVC
Razor Syntax Deep Dive Day 1

Learn @Model, @foreach, @if, Tag Helpers like asp-for, asp-action, asp-controller. Understand the difference between ViewBag, ViewData, and strongly typed models. Learn @Html.DisplayFor() vs Tag Helpers.

Resources → Razor Syntax Reference (docs.microsoft.com)
EF Core — Code First Day 2

Code First means you write C# classes → EF generates the database. Learn: DbContext, DbSet<T>, Data Annotations ([Required], [MaxLength], [ForeignKey]). Run Add-Migration InitialCreate then Update-Database. Understand the Migrations folder — never delete it.

Resources → EF Core Getting Started (Microsoft Docs)
LINQ Essentials Day 3

You'll use LINQ constantly. Master: .Where(x => x.UserId == userId), .ToList(), .FirstOrDefault(), .OrderByDescending(), .Any(), .Count(). Also learn .Include() for eager loading related entities — essential for habits + logs.

Resources → LINQ in C# — dotnetcurry.com
🔨 THEN BUILD 4 days
Design DB Schema Day 4

Create three models: ApplicationUser (extends IdentityUser), Habit (Id, Name, UserId, CreatedAt, Color, IsActive), HabitLog (Id, HabitId, LogDate, IsCompleted). Set up relationships: User → many Habits, Habit → many HabitLogs. Add FK with [ForeignKey("HabitId")] and navigation properties.

public class Habit {
  public int Id { get; set; }
  public string Name { get; set; }
  public string UserId { get; set; }
  public DateTime CreatedAt { get; set; }
  public ICollection<HabitLog> Logs { get; set; }
}
Full Habit CRUD Day 4–5

Build HabitsController with Index (list), Create (GET + POST), Edit (GET + POST), Delete (POST with anti-forgery token). Use IActionResult return types. Redirect with RedirectToAction(). Add model validation with ModelState.IsValid. Use _context.SaveChangesAsync().

Daily Check-In Grid Day 6

Build the exact grid from your screenshot. Create a ViewModel: HabitGridViewModel with habits list and last 7 dates. In the controller, query HabitLogs for each habit for the date range. Pass to view. In Razor, loop habits as rows, dates as columns, render ✓ if log exists. POST endpoint to toggle a log entry.

// Check if log exists for date
var log = await _context.HabitLogs
  .FirstOrDefaultAsync(l =>
    l.HabitId == habitId &&
    l.LogDate.Date == date.Date);
Streak Calculation Day 7

Write a service method GetStreakCount(int habitId). Query logs ordered by date descending. Iterate — count consecutive days where IsCompleted is true, starting from today. Break the loop when a gap is found. Return the count. This is pure C# logic — great practice.

✅
WEEK 1 DELIVERABLE

Working habit app with grid view (matching your screenshot), streak counts, SQL Server DB with proper schema, full CRUD. No login yet — all habits are open.

02
WEEK TWO · DAYS 8–14

Authentication
Identity + Google Login

Transform your open app into a multi-user system. Each person gets their own private habit data. Learn the most important skill in web dev — auth.

📘 LEARN FIRST 3 days
ASP.NET Identity System Day 8

Identity is a complete auth system. Learn: scaffold Identity pages (dotnet aspnet-codegenerator identity), IdentityDbContext, UserManager<T>, SignInManager<T>. Understand the tables it creates: AspNetUsers, AspNetRoles, AspNetUserClaims. Learn [Authorize] attribute and how to get current user ID with User.FindFirstValue(ClaimTypes.NameIdentifier).

OAuth 2.0 — The Concept Day 9

OAuth 2.0 is an authorization framework. Flow: your app → redirects user to Google → user grants permission → Google sends back a code → your app exchanges code for access token + refresh token. Key concepts: Client ID, Client Secret, Redirect URI, Scopes. Watch "OAuth 2.0 explained in 5 minutes" on YouTube first, then read the ASP.NET docs.

Google Cloud Console Setup Day 9

Go to console.cloud.google.com → New Project → APIs & Services → Credentials → Create OAuth 2.0 Client ID. Set Application Type to "Web Application". Add Authorized Redirect URIs: https://localhost:7xxx/signin-google. Copy Client ID and Client Secret. Store them in appsettings.json or better — User Secrets (dotnet user-secrets set) to avoid committing secrets to Git.

// Program.cs
builder.Services.AddAuthentication()
  .AddGoogle(opts => {
    opts.ClientId = config["Google:ClientId"];
    opts.ClientSecret = config["Google:Secret"];
  });
Claims & User Identity Day 10

A Claim is a key-value pair about the user. After login, you get claims like name, email, picture URL. Learn ClaimsPrincipal, HttpContext.User. In controllers, always filter data by: var userId = User.FindFirstValue(ClaimTypes.NameIdentifier); then _context.Habits.Where(h => h.UserId == userId). This is the foundation of multi-user security.

🔨 THEN BUILD 4 days
Scaffold & Wire Identity Day 10–11

Run the Identity scaffolder. Update ApplicationDbContext to extend IdentityDbContext<ApplicationUser>. Add new migration. Register services in Program.cs: AddDefaultIdentity, AddEntityFrameworkStores. Add app.UseAuthentication() before app.UseAuthorization() — order matters! Update _Layout.cshtml to show login/logout in navbar.

Add Google Login Button Day 11

In the Login.cshtml (scaffolded), external providers appear automatically as buttons if configured correctly. Style the Google button with proper branding. Test the full flow: click Google → Google consent screen → redirect back → user is created in AspNetUsers. Check the DB — you should see the new user row with Google's external ID stored in AspNetUserLogins.

Secure All Controllers Day 12

Add [Authorize] to HabitsController. Update every query to filter by UserId. On Create, set habit.UserId = User.FindFirstValue(ClaimTypes.NameIdentifier). On Edit/Delete, always verify the habit belongs to the current user before modifying — otherwise any logged-in user can delete another's data. This is called an IDOR vulnerability — prevent it from day 1.

// Always verify ownership!
var habit = await _context.Habits
  .FirstOrDefaultAsync(h =>
    h.Id == id && h.UserId == userId);
if (habit == null) return Forbid();
User Profile Display Day 13–14

Show Google profile picture and name in the navbar. Get user info from claims: User.FindFirstValue("picture") for Google avatar, User.FindFirstValue(ClaimTypes.Name) for name. If picture claim isn't available, fetch from UserManager.GetUserAsync(User). Store profile picture URL in your ApplicationUser model.

✅
WEEK 2 DELIVERABLE

Multi-user app with Google login working. Each user sees only their habits. Navbar shows profile pic and name. All routes are protected. IDOR security in place.

WEEK THREE · DAYS 15–21

Google Fit
Calories + Distance Goals

The hardest week. Integrate a real fitness API, handle OAuth token storage, parse complex JSON, and surface real-time health data in your app.

📘 LEARN FIRST 3 days
Google Fit REST API Overview Day 15

Read the Google Fit REST API docs. Understand the concept of data sources and data points. Key endpoints: users.dataset.aggregate is what you'll use most. It accepts a JSON body with startTimeMillis, endTimeMillis, and aggregateBy (dataTypeName). Use Postman to test before writing code — authenticate with your token and hit the endpoint manually first.

// Aggregation request body
{
  "aggregateBy": [{
    "dataTypeName":
    "com.google.calories.expended"
  }],
  "startTimeMillis": ...,
  "endTimeMillis": ...
}
OAuth Scopes for Fitness Day 15

Add fitness scopes when configuring Google auth. You need: https://www.googleapis.com/auth/fitness.activity.read for calories and steps, https://www.googleapis.com/auth/fitness.location.read for distance. In Program.cs, add opts.Scope.Add("..."). Also set opts.SaveTokens = true — this stores access and refresh tokens in the auth cookie so you can retrieve them later.

Storing OAuth Tokens in DB Day 16

Tokens from SaveTokens = true are in the cookie — not persistent. For server-side calls, store tokens in your DB. Use the ITokenStore pattern or save to your ApplicationUser table. Add columns: GoogleAccessToken, GoogleRefreshToken, TokenExpiry. In OnCreatingTicket callback, extract and store them. Refresh tokens don't expire unless revoked — use them to get new access tokens silently.

HttpClient in C# Day 16

Learn to make HTTP requests from your server. Always use IHttpClientFactory — inject it in the service constructor. Create named clients. Use client.DefaultRequestHeaders.Authorization = new AuthenticationHeaderValue("Bearer", accessToken). Use PostAsJsonAsync(), read response with response.Content.ReadAsStringAsync(), then deserialize with JsonSerializer.Deserialize<T>().

🔨 THEN BUILD 4 days
GoogleFitService Class Day 17–18

Create Services/GoogleFitService.cs. Inject IHttpClientFactory and ApplicationDbContext. Write GetTodayCaloriesAsync(string userId) and GetTodayDistanceKmAsync(string userId). Fetch stored token, make POST request to Fit API, parse the nested JSON response. The response has bucket → dataset → point → value structure — use JsonDocument to navigate it.

// Fit API response structure
bucket[0].dataset[0].point[0]
  .value[0].fpVal // calories as float

// Distance in meters, convert:
var km = meters / 1000.0;
Token Refresh Logic Day 18

Access tokens expire after 1 hour. Before calling the Fit API, check TokenExpiry < DateTime.UtcNow. If expired, use the refresh token: POST to https://oauth2.googleapis.com/token with grant_type=refresh_token, your refresh token, client ID, and secret. Parse the response for a new access token and update the DB. Wrap this in a helper method called before every API call.

Fitness Goals System Day 19

Add a FitnessGoal model: UserId, DailyCalorieTarget, DailyDistanceTargetKm. Create a GoalsController with a settings page where users set their targets. On the dashboard, compare actual vs target and calculate percentage. Show a progress bar: actualCalories / targetCalories * 100. Color it green if > 80%, yellow if 50–80%, red if < 50%.

Error Handling for API Failures Day 20–21

The Fit API WILL fail — no phone data, revoked permissions, network errors. Wrap all Fit calls in try-catch. Return a FitResult object with a success flag and error message. On the UI, show a friendly message: "Connect your Google Fit app to see data" instead of crashing. Log errors with ILogger<T> for debugging. Add a "Reconnect Google Fit" button that re-triggers the OAuth flow with fitness scopes.

✅
WEEK 3 DELIVERABLE

Dashboard shows real-time calories burned and km walked from Google Fit. User can set daily goals. Progress bars show completion. Token refresh works silently. Graceful error handling.

WEEK FOUR · DAYS 22–30

Dashboard
Charts, Polish & Deploy

Bring it all together. Build the full dashboard, add visualizations, optimize performance, and optionally deploy to Azure or Railway.

📘 LEARN FIRST 2 days
Chart.js Integration Day 22

Chart.js is a JS library — include via CDN. The key challenge is passing C# data to JS. In Razor: var labelsJson = Json.Serialize(labels); then in a script tag: const labels = @Html.Raw(labelsJson);. Build bar charts for weekly habit completion, line charts for streak history, doughnut charts for goal achievement. Always put chart config in wwwroot/js/dashboard.js.

// C# → JS data bridge in Razor
@{ var data = Json.Serialize(model.WeeklyData); }
<script>
  const chartData = @Html.Raw(data);
</script>
ViewModels & Partial Views Day 23

Your dashboard aggregates data from habits, logs, and Fit API. Create a DashboardViewModel that holds everything. Build Partial Views for reusable components: _HabitGrid.cshtml, _FitnessCard.cshtml, _StreakBadge.cshtml. Use @await Html.PartialAsync("_HabitGrid", model). This keeps your main view clean and your components reusable.

🔨 THEN BUILD 7 days
Main Dashboard Layout Day 23–24

Design dashboard with Bootstrap 5 grid. Layout: top row = summary stats (streak, habits done today, calories, km). Middle = habit check-in grid (the table from your screenshot). Bottom = charts row. Use Bootstrap cards for each section. Load Fit data asynchronously with a small loading spinner — don't block the whole page if API is slow.

Weekly Completion Charts Day 24–25

For each habit, calculate completion rate for last 7 days. Chart.js bar chart: X axis = habit names, Y axis = % completion. Add a line chart showing daily total habits completed over the last 30 days. Use the color from your Habit model for bar colors — personalized charts feel premium. Add tooltips showing exact completion counts.

Streak Leaderboard & Badges Day 25–26

Show habits ordered by current streak. Award emoji badges: 🔥 3+ days, ⚡ 7+ days, 💎 30+ days. These are calculated on the server and passed to the view. Also show "best ever streak" per habit — query all logs and find the longest consecutive run. Store this in a computed column or recalculate on demand (fine for now).

Mobile Responsive + Final Polish Day 27–28

Test on mobile — Bootstrap handles most of it, but the habit grid needs horizontal scroll on small screens: overflow-x: auto on the table container. Add a dark/light mode toggle using localStorage and CSS variables. Add page transitions with a simple fade using Bootstrap's fade class. Clean up all unused code, fix spelling, ensure consistent button styles.

Optional: Deploy to Azure Day 29–30

Deploy to Azure App Service (free tier) + Azure SQL Database. Use the Visual Studio "Publish" wizard or GitHub Actions. Update Google OAuth redirect URIs to your production URL. Move secrets to Azure App Settings (Environment Variables) — never in code. Add ASPNETCORE_ENVIRONMENT=Production. Alternatively, Railway.app is simpler for beginners.

✅
WEEK 4 DELIVERABLE

Complete, polished app — habit grid, streak badges, weekly charts, fitness goals, mobile responsive. Ready to add to your portfolio and GitHub. Optionally live on Azure.

// BEGINNER RULES TO LIVE BY
01
Learn → Code Same Day. Never watch more than 1 hour of tutorials without opening Visual Studio and coding what you just learned. Passive watching = forgetting.
02
Git from Day 1. git init before your first line. Commit every night. You WILL break your app — Git is your undo button.
03
Never store secrets in code. Use dotnet user-secrets for development. Never commit appsettings.json with real credentials to GitHub.
04
Break problems down. If something feels too big, write in English what it needs to do, step by step. Then code each step. A working ugly solution beats a perfect one that doesn't exist.
05
Google Fit is hard — give it time. Week 3 will be frustrating. Use Postman to test API calls first. Read the error messages carefully. Token issues account for 80% of Fit API problems.
06
Get it working, then make it pretty. Week 4 is for polish. Weeks 1–3 are about functional code. A working feature beats a beautiful broken one every time.
